What I Built: Isagawa

Isagawa is an agentic spec-driven development environment. It turns high-level intent into structured development workflows, applies domain-aware controls during execution, and produces reviewable software artifacts with explicit validation evidence.

Specification Externalize intent into structured, declarative formats before execution begins
Domain Adaptation Configure execution for different verticals without rewriting the runtime
Governed Execution Enforce bounded work units with explicit gates throughout the workflow
Validation Independent verification of outputs against declared invariants
Review Structured review workflows for generated artifacts before delivery
Workflow Tracking Persistent state and progress tracking across sessions

I designed and implemented the system architecture, workflow runtime, evaluation mechanisms, and applied platforms.

The Problem and Design Thesis

Coding agents can lose procedural context mid-task, reinterpret instructions as conversation history grows, and report completion without independent evidence that the work is correct.

Advisory prompts Instructions provided as context, not as enforceable constraints — the model can drift from them without any mechanism noticing
Fragile conversational state Prior decisions, corrections, and constraints fade as context windows compress or conversations restart
Self-reported completion The same model that performed the work is the only judge of whether it succeeded — no independent verification layer

Specifications alone are not enough. A spec can declare what should happen, but without a runtime that enforces bounded execution, tracks state outside the model's context, and validates outputs independently, the gap between intent and result remains open.

My approach: treat agent reliability as a software runtime and verification problem — not only a prompting problem.

High-Level Architecture

Isagawa organizes AI-assisted development around four concepts that separate intent from execution and require independent evidence before work is considered complete.

Structured Intent Developer goals are captured in declarative specifications before any execution begins — the model works from an externalized plan, not improvised conversation
Domain-Aware Workflow Execution adapts to the target domain's conventions, interfaces, and constraints without rewriting the underlying runtime
Governed Execution Work proceeds in bounded units with explicit gates — progress is checkpointed outside the model's context so it survives session boundaries
Independent Validation and Review Outputs are verified against declared invariants by mechanisms separate from the model that produced them — completion requires evidence, not self-report
Externalized State Workflow state is persisted outside the model's context window, supporting inspection, recovery, and continuity across sessions
Bounded Work Execution is divided into explicit units with defined entry and exit criteria, reducing ambiguity and enabling incremental validation
Tool-Aware Controls The runtime uses tool-level instrumentation to enforce workflow rules — controls operate alongside the model, not as advisory context within it
Explicit Gates Progression between workflow stages requires passing defined checks — the system does not advance on self-reported completion alone

Key principle: the model does not serve as the only source of state or the only judge of completion.

Governed Workflow

Execution proceeds in bounded work units. Each unit has an explicit entry condition, a defined action, and a completion gate. Progress is checkpointed outside the model's context window so it persists across session boundaries. A task runner iterates through the unit queue, selecting an appropriate model per task and advancing only when the gate passes.

Why it matters: bounded units prevent open-ended drift. If the model loses context or restarts, the workflow resumes from the last checkpoint — not from scratch.

Independent Validation

Each deliverable passes through gate checks defined separately from the code that produced it. Gates range from structural checks (does the file exist, does it parse, do imports resolve) to live behavioral tests (does the workflow run against a real application and produce the expected outcome). A failing gate blocks progression — the system does not advance on the model's self-report.

Why it matters: the model that wrote the code is not the sole judge of whether it works. Verification is independent of generation.

Controlled Delivery

Generated changes are isolated to feature branches. Each batch of work produces a reviewable diff before merging. Commit history tracks what was built, what was tested, and what was validated — the full chain from intent to artifact is inspectable after the fact.

Why it matters: AI-generated code enters the codebase through the same review pathway as human-written code. Nothing merges unexamined.

Pending verified figure: branches and review cycles

Evaluation Routing

The system selects model and verification strategy based on workload type. Build tasks that require architectural reasoning route to higher-capability models; structural tests route to faster, cheaper models. Invalid or malformed inputs are rejected before execution begins rather than failing mid-workflow.

Why it matters: not every task needs the same model or the same depth of verification. Routing reduces cost and latency without sacrificing quality where it matters most.

Case Study: Multi-Interface QA Platform

A cross-interface QA automation platform built and validated through Isagawa, covering browser, REST API, database, and SOAP service testing under a unified architectural model.

Problem

Enterprise workflows required coordinated browser, API, database, and SOAP validation without duplicating architecture across tools. Each interface has different timing characteristics, data formats, and failure modes — but the test logic and governance model needed to stay consistent across all of them.

My Role

I designed the framework architecture, cross-interface contracts, dependency model, test-data strategy, and AI-assisted development workflow. Every layer — from interface adapters to the orchestration and validation pipeline — was designed and implemented by me, using Isagawa to govern the build process itself.

Solution

One coherent platform model across four interfaces, each sharing a common architectural contract while adapting to the specific capabilities of its transport layer. The platform is built through Isagawa's governed workflow — each interface slice is decomposed into bounded tasks, built incrementally, and validated against defined gates before merging.

Result

Browser, REST API, and SQL Server interfaces are built and validated end to end under the shared contract; the SOAP interface is the next slice on the same architecture. Isagawa governed the build itself — each interface was decomposed into bounded tasks and passed through independent gate checks before merging.

Coverage

Browser Automation Complete — Selenium-based interface with page objects, component abstractions, role-scoped workflows, and structural and live end-to-end test suites
REST API Complete — request/response interface with typed API objects, task workflows, and test suites covering endpoint validation and data consistency
SQL Server Complete — direct-database interface with data objects, query workflows, role layers, and structural and live end-to-end test suites
SOAP / XML Services Planned — architecture designed and contracts defined; implementation scheduled as the next platform slice

Unified Governance

All four interfaces share a common architectural contract that enforces consistent layering, naming, and dependency rules. Each interface adapts the contract to its transport while preserving the same structural conventions — a browser page object and a REST API object follow the same compositional pattern. The contract is enforced at build time through Isagawa's gate system, not through manual review alone.

Engineering Decisions and Trade-offs

Each of these decisions trades a benefit for a concrete operational cost. None is claimed to be universally optimal — each is a fit for the problem the runtime is solving.

Decision Reason Operational Trade-off / Invariant
Persist workflow state outside chat Supports inspection, recovery, continuity Adds a persistence layer to maintain; state must survive session boundaries
Bound work into explicit units Reduces ambiguity, improves validation Requires upfront task decomposition before execution
Separate mechanical and semantic checks Different invariants need different verification methods Semantic checks remain model-assisted and probabilistic; mechanical checks are deterministic
Use controlled integration Keeps generated changes reviewable before merge Adds a review step; trades some autonomy for auditability
Adapt by domain Reusable runtime without hardcoding one vertical Domain specs must be authored per vertical; runtime stays constant
