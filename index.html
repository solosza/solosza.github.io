What I Built: Isagawa

Isagawa is an agentic spec-driven development environment. It turns high-level intent into structured development workflows, applies domain-aware controls during execution, and produces reviewable software artifacts with explicit validation evidence.

Specification Externalize intent into structured, declarative formats before execution begins
Domain Adaptation Configure execution for different verticals without rewriting the runtime
Governed Execution Enforce bounded work units with explicit gates throughout the workflow
Validation Independent verification of outputs against declared invariants
Review Structured review workflows for generated artifacts before delivery
Workflow Tracking Persistent state and progress tracking across sessions

I designed and implemented the system architecture, workflow runtime, evaluation mechanisms, and applied platforms.

The Problem and Design Thesis

Coding agents can lose procedural context mid-task, reinterpret instructions as conversation history grows, and report completion without independent evidence that the work is correct.

Advisory prompts Instructions provided as context, not as enforceable constraints — the model can drift from them without any mechanism noticing
Fragile conversational state Prior decisions, corrections, and constraints fade as context windows compress or conversations restart
Self-reported completion The same model that performed the work is the only judge of whether it succeeded — no independent verification layer

Specifications alone are not enough. A spec can declare what should happen, but without a runtime that enforces bounded execution, tracks state outside the model's context, and validates outputs independently, the gap between intent and result remains open.

My approach: treat agent reliability as a software runtime and verification problem — not only a prompting problem.

High-Level Architecture

Isagawa organizes AI-assisted development around four concepts that separate intent from execution and require independent evidence before work is considered complete.

Structured Intent Developer goals are captured in declarative specifications before any execution begins — the model works from an externalized plan, not improvised conversation
Domain-Aware Workflow Execution adapts to the target domain's conventions, interfaces, and constraints without rewriting the underlying runtime
Governed Execution Work proceeds in bounded units with explicit gates — progress is checkpointed outside the model's context so it survives session boundaries
Independent Validation and Review Outputs are verified against declared invariants by mechanisms separate from the model that produced them — completion requires evidence, not self-report
Externalized State Workflow state is persisted outside the model's context window, supporting inspection, recovery, and continuity across sessions
Bounded Work Execution is divided into explicit units with defined entry and exit criteria, reducing ambiguity and enabling incremental validation
Tool-Aware Controls The runtime uses tool-level instrumentation to enforce workflow rules — controls operate alongside the model, not as advisory context within it
Explicit Gates Progression between workflow stages requires passing defined checks — the system does not advance on self-reported completion alone

Key principle: the model does not serve as the only source of state or the only judge of completion.
